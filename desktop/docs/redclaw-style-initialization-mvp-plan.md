# RedClaw 首次风格初始化 MVP 方案

## 1. 目标

这份文档定义 `RedClaw` 首次风格初始化的最小实现版。

目标不是一次做完整心理测试系统，而是先上线一个可用的 `MVP`：

- 总题量控制在 `10` 题左右
- 以 `滑杆题` 为主
- 只保留最少的分类题
- 只保留最少的 `A/B` 文案题
- 结果能够直接生成首版长期档案

这个 `MVP` 要优先解决三个问题：

1. 这个空间到底更偏内容还是更偏商业
2. 这个空间更依赖人设还是品牌
3. 这个空间的文案到底应该怎么写

同时，这个 `MVP` 还需要明确一个简单稳定的交互载体：

- 用户第一次在当前空间与 `RedClaw` 对话时
- `RedClaw` 主动发出一个风格初始化入口链接
- 用户点击后进入一个 `全屏独立页面`
- 问卷使用 `一题一页` 的方式完成

## 2. MVP 设计原则

### 2.1 优先采集“权重”，不优先采集“复杂分类”

因为多数真实空间都不是纯单一模式，所以 `MVP` 的主输入应该是滑杆，而不是树状分支。

### 2.2 先拿到足够可执行的信息，不追求一次性完整理解用户

`MVP` 不做完整多层心理测试。

只要能生成：

- 基础经营方向
- 基础品牌方向
- 基础写作方向
- 基础协作方向

就可以上线。

### 2.3 题目必须能直接映射到档案

每一题都必须清楚知道写到哪里：

- `Soul.md`
- `user.md`
- `CreatorProfile.md`
- `style-profile.json`
- 后续 overlay skill 摘要

### 2.4 入口必须发生在第一次真实对话里

不要把初始化藏在设置页，也不要让用户先离开对话再去找入口。

推荐的 `MVP` 入口方式是：

- 用户第一次进入某个空间的 `RedClaw` 对话
- `RedClaw` 先发一条欢迎消息
- 同一条消息里附带一个明显的“定义这个空间的创作风格”入口

这样有三个好处：

1. 用户能理解这是当前空间专属设置，不是全局设置
2. 问卷和后续创作任务是在同一个上下文里发生的
3. 如果用户先跳过，后面也能在当前对话里继续补做

### 2.5 问卷必须是全屏独立流程，不嵌在聊天气泡里

`MVP` 阶段不建议把滑杆题直接做成聊天里的交互气泡。

推荐改成：

- 聊天里只放入口卡片或链接
- 点击后进入一个全屏独立页面
- 每道题一页
- 顶部展示进度

原因：

1. 滑杆题和 `A/B` 题在聊天流里会显得很乱
2. 一题一页更像真正的“风格测试”，用户负担更低
3. 后续要加说明文案、返回上一步、保存进度时，更容易扩展

## 3. MVP 总题量

建议题量：

- `7` 个滑杆题
- `2` 个分类题
- `1` 个 `A/B` 文案题

总计 `10` 题。

这是最小但仍然有执行价值的版本。

## 3.1 MVP 交互形态

推荐交互形态固定为：

1. `第一次对话触发入口`
2. `点击进入全屏页面`
3. `一题一页`
4. `顶部进度条`
5. `底部主按钮：下一题 / 完成`
6. `支持返回上一题`
7. `支持关闭并稍后继续`

这套形态在 `MVP` 阶段已经足够，不需要额外做复杂分栏或聊天内嵌控件。

## 3.2 首次对话触发规则

建议触发条件：

- 当前空间 `onboarding` 尚未完成
- 用户第一次在这个空间打开 `RedClaw` 并发送消息，或首次进入空白对话页

推荐行为：

- `RedClaw` 正常欢迎用户
- 然后追加一条初始化提示

示例文案：

> 在开始长期创作前，先定义一下这个空间的经营方向和写作风格。  
> 这会影响我后续怎么帮你写内容、定调性和安排转化。  
> [开始定义这个空间]

如果用户暂时不点：

- 当前对话可以继续
- 但后续仍保留一个未完成状态
- 后面可以再次提醒或从对话顶部入口继续

## 3.3 全屏独立页面规则

点击入口后，进入一个专用全屏页面，而不是弹窗。

推荐结构：

- 顶部：标题、当前空间名、进度、关闭按钮
- 中间：当前题目
- 底部：上一题 / 下一题 / 完成

页面特征：

- 全屏覆盖当前 `RedClaw` 页面
- 保持当前空间上下文
- 完成后回到原对话
- 完成结果回流到当前对话线程

## 3.4 一题一页规则

每一页只展示一题，不把多题堆在同一屏。

每题页面结构建议固定：

1. 题目标题
2. 一句简短解释
3. 主交互控件
4. 当前选择的即时文本解释
5. 下一步按钮

这样做的原因：

- 用户更容易专注
- 滑杆题的感知更清晰
- `A/B` 题不会和其他题互相干扰
- 后续埋点和分析更简单

## 3.5 进度与中断恢复

`MVP` 必须支持最基本的中断恢复。

至少要保存：

- 当前题号
- 已回答题目
- 当前所有滑杆值
- 已选分类项
- `A/B` 题选择

关闭后再次点击入口时：

- 默认从上次未完成的题继续

完成后：

- 状态标记为 `completed`
- 入口卡片不再默认主动弹出
- 对话里追加一条“风格定义已完成”的摘要消息

## 4. MVP 题目清单

## 4.1 题 1：内容 vs 商业

题型：

- 滑杆

页面说明：

- 一页只放这个滑杆
- 滑动时实时显示当前值解释

题目：

`这个空间整体更偏内容，还是更偏商业？`

滑杆两端：

- `0 = 纯内容导向`
- `100 = 纯商业导向`

用途：

- 定义整个空间的总经营重心

落盘：

- `style-profile.json.workspaceMission.contentVsCommerce`
- `user.md` 摘要
- `CreatorProfile.md` 经营方向

## 4.2 题 2：人设 vs 品牌

题型：

- 滑杆

页面说明：

- 实时显示“当前更偏人设驱动 / 品牌驱动”的解释文本

题目：

`这个空间更依赖经营你这个人，还是经营品牌本身？`

滑杆两端：

- `0 = 完全品牌驱动`
- `100 = 完全人设驱动`

用途：

- 判断长期信任来源

落盘：

- `style-profile.json.businessModel.personaVsBrand`
- `CreatorProfile.md`

## 4.3 题 3：一致性 vs 爆发力

题型：

- 滑杆

页面说明：

- 实时展示“更稳 / 更猛”的总结文案

题目：

`这个空间更应该追求长期一致性，还是短期爆发力？`

滑杆两端：

- `0 = 一致性优先`
- `100 = 爆发力优先`

用途：

- 决定选题和表达策略更稳还是更猛

落盘：

- `style-profile.json.brandStrategy.consistencyVsVirality`
- `CreatorProfile.md`

## 4.4 题 4：专业感 vs 亲近感

题型：

- 滑杆

页面说明：

- 用一句辅助语解释这道题决定的是“账号给人的第一印象”

题目：

`你更希望账号给人的感觉是专业判断，还是亲近自然？`

滑杆两端：

- `0 = 亲近自然`
- `100 = 专业判断`

用途：

- 决定账号气质和表达姿态

落盘：

- `style-profile.json.audienceModel.authorityPosture`
- `style-profile.json.writingPreferences.authorityLevel`
- `CreatorProfile.md`

## 4.5 题 5：冷静克制 vs 情绪感染

题型：

- 滑杆

页面说明：

- 页面底部显示“更冷静 / 更有感染力”的即时反馈

题目：

`文案整体更应该冷静克制，还是更有情绪感染力？`

滑杆两端：

- `0 = 极冷静`
- `100 = 强感染`

用途：

- 决定情绪浓度

落盘：

- `style-profile.json.writingPreferences.emotionalTemperature`
- `user.md` 偏好摘要

## 4.6 题 6：弱转化 vs 强转化

题型：

- 滑杆

页面说明：

- 实时反馈“转化表达偏隐性 / 偏显性”

题目：

`内容里的转化表达应该更隐性，还是更显性？`

滑杆两端：

- `0 = 基本不直接转化`
- `100 = 明确推动行动`

用途：

- 决定文案里的销售显性度与 CTA 强度

落盘：

- `style-profile.json.writingPreferences.salesExplicitness`
- `style-profile.json.writingPreferences.ctaIntensity`

## 4.7 题 7：框架拆解 vs 故事表达

题型：

- 滑杆

页面说明：

- 实时反馈“正文偏框架 / 偏故事”

题目：

`文案更应该偏框架拆解，还是偏故事化表达？`

滑杆两端：

- `0 = 故事表达`
- `100 = 框架拆解`

用途：

- 决定正文组织方式

落盘：

- `style-profile.json.writingPreferences.structurePreference`
- `style-profile.json.writingPreferences.storyRatio`

## 4.8 题 8：经营模式

题型：

- 单选

页面说明：

- 用大卡片单选，不用下拉框

题目：

`这个空间目前最接近哪一种经营方式？`

选项：

- `人设带货`
- `品牌带货`
- `高客单服务转化`
- `纯内容账号`

说明：

`MVP` 先不做更细的单品爆款、店铺矩阵等分支。

用途：

- 给空间定一个最基础的策略类型

落盘：

- `style-profile.json.businessModel.primaryModel`
- `CreatorProfile.md`

## 4.9 题 9：账号角色

题型：

- 单选

页面说明：

- 用大卡片单选，并给每个角色一句解释

题目：

`你希望受众主要把你视为什么角色？`

选项：

- `专业顾问`
- `有经验的过来人`
- `真实试错者`
- `品牌主理人`

用途：

- 约束账号的人物姿态

落盘：

- `style-profile.json.audienceModel.rolePosition`
- `CreatorProfile.md`
- `user.md`

## 4.10 题 10：A/B 开头偏好

题型：

- `A/B` 文案题

页面说明：

- 左右或上下展示两张文案卡片
- 只允许选一个
- 选中后再进入完成页

题目：

`下面两种开头，你更愿意长期采用哪一种？`

A：

> 很多人以为卖不动，是流量不够。  
> 但多数时候，问题根本不在流量。

B：

> 这周我重新看了 37 条转化不错的内容。  
> 最后发现，真正影响成交的不是你想的那个点。

用途：

- 在 `MVP` 阶段，只保留一组最关键的文案偏好题，用于区分“强判断钩子”与“观察式开头”

落盘：

- `style-profile.json.writingPreferences.hookStrength`
- `style-profile.json.writingPreferences.openingStyle`
- 后续 overlay skill 摘要

## 5. 结果映射规则

## 5.1 `Soul.md`

`MVP` 阶段只写一条简单协作风格默认值，不在初始化问卷里额外占题。

默认建议：

- `高执行 + 强结构 + 直接反馈`

后续再通过真实对话纠偏更新。

原因：

- 协作风格虽然重要，但不是当前 `MVP` 的核心阻塞项
- 优先把题量留给经营方向和文案风格

## 5.2 `user.md`

`MVP` 阶段写入：

- 核心经营倾向摘要
- 账号角色
- 风格偏好摘要

示例：

- 该空间内容与商业并行，但商业略优先
- 偏人设驱动
- 偏专业判断、冷静表达、框架拆解

## 5.3 `CreatorProfile.md`

`MVP` 阶段写入：

- 经营模式
- 信任来源
- 一致性/爆发力取向
- 角色定位
- 长期风格方向

## 5.4 `style-profile.json`

`MVP` 阶段作为唯一真源，至少需要这些字段：

```json
{
  "workspaceMission": {
    "contentVsCommerce": 0
  },
  "businessModel": {
    "personaVsBrand": 0,
    "primaryModel": ""
  },
  "audienceModel": {
    "authorityPosture": 0,
    "rolePosition": ""
  },
  "brandStrategy": {
    "consistencyVsVirality": 0
  },
  "writingPreferences": {
    "emotionalTemperature": 0,
    "salesExplicitness": 0,
    "ctaIntensity": 0,
    "structurePreference": "",
    "storyRatio": 0,
    "hookStrength": 0,
    "openingStyle": ""
  },
  "collaborationPreferences": {
    "executionStyle": "high_execution_structured_direct"
  }
}
```

## 5.5 overlay skill 摘要

`MVP` 阶段可以先不真正生成独立 overlay skill 文件。

第一步只需要在 `style-profile.json` 基础上生成一段可注入摘要，供写作时使用。

示例：

- 开头优先判断先行 / 观察式开头
- 整体偏专业 / 亲近
- 偏冷静 / 偏情绪
- 转化表达偏强 / 偏弱
- 正文偏框架拆解 / 偏故事推进

这样能先绕开 skill 文件自动生成的实现复杂度。

## 6. MVP 交互顺序

建议顺序：

1. 内容 vs 商业
2. 人设 vs 品牌
3. 一致性 vs 爆发力
4. 专业感 vs 亲近感
5. 冷静克制 vs 情绪感染
6. 弱转化 vs 强转化
7. 框架拆解 vs 故事表达
8. 经营模式
9. 账号角色
10. A/B 开头偏好

这个顺序的优点：

- 前 7 题连续完成，体验统一
- 后 2 题快速定模式
- 最后一题用真实文案样例收尾，增强“像风格测试”的感觉

## 6.1 完成后的回流方式

用户完成最后一题后，页面不直接静默关闭。

建议顺序：

1. 展示一页总结摘要
2. 用户点击“确认采用”
3. 返回当前 `RedClaw` 对话
4. 对话里出现一条系统摘要消息

摘要消息示例：

> 已完成这个空间的风格定义。  
> 当前倾向：商业 60 / 内容 40，人设 70 / 品牌 30，偏专业、偏冷静、偏框架拆解。  
> 我会按这份方向继续协作，后续你也可以随时纠偏。

## 7. MVP 结果确认页

完成后必须展示一页确认摘要，而不是直接结束。

建议展示：

- 内容/商业：`内容 40 / 商业 60`
- 人设/品牌：`人设 70 / 品牌 30`
- 一致性/爆发力：`一致性 55 / 爆发力 45`
- 气质：`偏专业、偏冷静`
- 转化方式：`中等偏强`
- 结构方式：`偏框架拆解`
- 账号模式：`人设带货`
- 账号角色：`专业顾问`
- 开头方式：`偏强判断钩子`

并提供两个动作：

- `确认采用`
- `重新调整`

`MVP` 阶段先不做“只改单题”。

## 7.1 确认页的页面角色

确认页本身也是全屏流程的一页，不是弹窗。

它承担两个职责：

1. 把抽象答案翻译成人能看懂的风格总结
2. 作为最后一次兜底，避免用户误点完成

## 7.2 与聊天线程的关系

确认完成后，初始化页面关闭，用户回到原来的 `RedClaw` 对话。

这个对话线程里要保留三类信息：

- 初始化已完成状态
- 一条可读摘要
- 一个重新进入定义流程的入口

## 8. 为什么这是合理的 MVP

这个版本没有做：

- 完整多分支题库
- 大量 A/B 文案测试
- 滑杆后的含义校准题
- 动态追问
- overlay skill 自动文件生成

但它已经能解决最核心的问题：

1. 先判断空间经营方向
2. 再判断账号气质
3. 再判断转化风格
4. 最后用一组真实文案题确认写法

这足够支撑首版个性化初始化上线。

## 9. 后续升级路径

在 `MVP` 基础上，后面可以自然升级：

1. 增加滑杆校准题
2. 把 1 组 `A/B` 开头题扩展到 4-6 组
3. 从写作摘要升级到真正的 overlay skill
4. 接入后续真实创作纠偏更新流程

## 10. 推荐结论

如果目标是尽快实现首版可用初始化流程，这个 `MVP` 足够小，也足够有用。

建议先做这 10 题版本：

- `7` 个滑杆
- `2` 个分类题
- `1` 个 `A/B` 文案题

先把最小闭环跑通，再扩展复杂逻辑树。
